# 세상 부수기 첫 진입 프로필 선택 설계

> 작성일: 2026-07-17
>
> 상태: PM 작성 문서 승인 완료, 구현 승인 완료
>
> 변경 성격: 기존 프로필 UI의 진입 순서 개선, DB·Supabase 스키마 변경 없음

## 1. 결정 요약

프로필 기능이 켜진 기기의 첫 방문자는 게임을 조작하기 전에 시작 방식을 한 번 선택한다.

- `새 프로필 만들기`
- `내 프로필로 로그인`
- `게스트로 시작`

게스트 선택은 같은 기기에 기억해 다음 접속부터 게임으로 바로 들어간다. 유효한 로그인 세션이 있으면 선택 화면을 건너뛰고 해당 프로필로 시작한다. 로그아웃하면 게스트 선택 기록을 지우고 시작 방식 선택 화면을 다시 보여준다.

새 화면을 별도로 디자인하지 않는다. 현재 프로필 전체 화면의 색상, 크기, 글자, 버튼, 여백, 모달 동작을 그대로 재사용하고 `첫 진입 필수 선택 상태`만 추가한다. 게임 HUD, 기록책, 무기 선택, Canvas 영역은 바꾸지 않는다.

이 결정은 `2026-07-16-player-profile-sync-design.md`의 `가입 안내에 막히지 않고 즉시 플레이`, `자동 가입 팝업을 표시하지 않는다`는 첫 진입 부분만 대체한다. 게스트 기록 분리, 새 프로필 기록 0 시작, 로그인·동기화·보안 원칙은 그대로 유지한다.

## 2. 목표와 제외 범위

### 목표

- 첫 방문자가 프로필 사용 여부를 게임 시작 전에 명확히 선택한다.
- 한 번 게스트를 선택한 기기에서는 같은 질문을 반복하지 않는다.
- 기존 로그인 세션은 지연 없이 복원한다.
- 로그아웃 후에는 다음 이용 방식을 다시 선택한다.
- 업데이트 안내보다 프로필 선택이 먼저 나온다.
- 기존 디자인과 모바일 조작감을 유지한다.
- 저장소나 네트워크 오류가 게스트 플레이를 막지 않는다.

### 제외 범위

- 프로필 ID·PIN·동기화 계약 변경
- Supabase 테이블, 함수, RLS, 기능 플래그 추가
- 게스트 기록을 프로필로 가져오기
- 게임 홈 화면, HUD, 기록책, 무기 바 재디자인
- 완성형 마이페이지와 업적 프로필 꾸미기
- 게스트 선택의 서버 동기화

## 3. 대안 비교와 채택

| 안 | 장점 | 약점 | 결정 |
|---|---|---|---|
| A. 기존 프로필 화면에 필수 선택 상태 추가 | 현재 디자인·접근성·폼을 재사용하고 변경 범위가 작음 | 시작 상태 구분이 추가됨 | 채택 |
| B. 별도 시작 화면 제작 | 화면 책임이 분리됨 | 모달·포커스·폼 진입 UI가 중복되고 디자인 차이가 생길 수 있음 | 제외 |
| C. 로그인 전 접속마다 선택 표시 | 저장 상태가 단순함 | 사용자가 확정한 `같은 기기에서 기억` 요구와 충돌 | 제외 |

## 4. 확정 UX

### 첫 진입 화면

- 제목: `어떻게 시작할까요?`
- 설명: `프로필로 시작하면 여러 기기에서 기록을 이어갈 수 있어요. 게스트로 시작하면 이 기기에만 기록돼요.`
- 첫 번째 행동: `새 프로필 만들기`
- 두 번째 행동: `내 프로필로 로그인`
- 세 번째 행동: `게스트로 시작`

세 버튼은 현재 프로필 화면의 버튼 구성과 우선순위 스타일을 사용한다. 새로운 색상, 아이콘, 카드, 일러스트를 추가하지 않는다.

### 필수 선택 상태

- 선택 전에는 닫기 버튼을 숨긴다.
- 배경 누르기, Escape, 브라우저 뒤로 가기로 닫히지 않는다.
- 뒤의 Canvas와 다른 UI는 `inert` 처리해 탭·터치·키보드 입력을 받지 않는다.
- 새 프로필 또는 로그인 하위 화면에서는 `뒤로`를 눌러 세 가지 선택으로 돌아갈 수 있다.
- 로그인이나 생성에 성공하면 필수 상태를 끝내고 해당 프로필로 게임을 시작한다.
- `게스트로 시작`을 누르면 선택을 기기에 저장하고 즉시 게임을 시작한다.

### 일반 프로필 화면

기록책에서 여는 기존 프로필 화면은 현재 동작을 유지한다.

- 닫기 버튼 유지
- 배경, Escape, 브라우저 뒤로 가기 닫기 유지
- 새 프로필·로그인·로그아웃·동기화 상태 UI 유지
- 필드, PIN 보기, 중복 확인, 저장 안내 문구 유지

### 로그아웃

- 정상 로그아웃이 완료되면 게스트 선택 기록을 삭제한다.
- 현재 열려 있는 프로필 화면을 필수 선택 상태로 전환한다.
- 사용자는 `게스트로 시작`, `새 프로필 만들기`, `내 프로필로 로그인` 중 하나를 다시 고른다.
- 미전송 기록이 있는 기존 로그아웃 보호 흐름은 그대로 먼저 처리한다.

## 5. 시작 순서

모달이 겹치지 않도록 시작 흐름을 한 곳에서 조정한다.

```text
게임 기본 화면 준비
  -> 업데이트 안내 자동 열기 보류
  -> 프로필 기능 플래그와 저장된 세션 확인
  -> 유효한 프로필 세션: 프로필 기록으로 시작
  -> 세션 없음 + 게스트 선택 기억: 게스트로 바로 시작
  -> 세션 없음 + 선택 기억 없음: 필수 선택 화면 표시
  -> 선택 완료
  -> 아직 보지 않은 업데이트 안내 표시
```

우선순위는 다음과 같다.

1. 운영자 임시 PIN을 받은 프로필의 강제 PIN 변경
2. 첫 진입 시작 방식 선택
3. 업데이트 안내
4. 게임의 일회성 조작 안내와 일반 알림

유효한 프로필 세션은 게스트 선택 기록보다 항상 우선한다. 프로필 로그인 또는 생성 성공 시 오래된 게스트 선택 기록이 있으면 삭제한다.

## 6. 로컬 상태 계약

- 키: `btw.profileEntry.v1`
- 허용 값: 정확히 `guest`
- 키가 없거나 값이 다르면 선택하지 않은 상태로 본다.
- 프로필 ID, PIN, 사용자 UUID, 게임 기록은 이 키에 저장하지 않는다.
- 로그인 세션은 기존 Supabase 세션 저장을 그대로 사용한다.

상태 전이:

| 현재 상태 | 사건 | 다음 상태 |
|---|---|---|
| 선택 없음 | 게스트 선택 | `guest` 저장, 게스트 플레이 |
| 선택 없음 | 생성·로그인 성공 | 키 삭제, 프로필 플레이 |
| `guest` | 재접속, 세션 없음 | 선택 화면 생략, 게스트 플레이 |
| `guest` | 유효한 세션 복원 | 키 삭제, 프로필 플레이 |
| 프로필 | 정상 로그아웃 | 키 삭제, 필수 선택 화면 |
| 어떤 상태든 | 저장소 읽기·쓰기 실패 | 현재 탭은 계속 진행, 다음 접속에서 다시 선택 가능 |

`localStorage`가 막힌 환경에서는 게스트 선택을 현재 탭에서만 인정한다. 저장 성공으로 가장하지 않으며 다음 접속에서 다시 물을 수 있다.

## 7. 구성요소 경계

### 첫 진입 결정기

작은 DOM 비의존 모듈이 세션 결과, 기능 플래그, 게스트 선택 기록을 받아 `player`, `guest`, `choose`, `fallback-guest` 중 하나를 결정한다. 로컬 키의 읽기·쓰기·삭제도 이 경계를 통과시켜 테스트 가능하게 만든다.

### 프로필 화면

기존 `PlayerProfileView`에 일반 열기와 필수 열기를 구분하는 모드를 추가한다. 폼과 검증 로직은 공유하며 필수 모드에서만 닫기 동작과 `게스트로 시작`을 다르게 처리한다.

### 시작 조정

`main.ts`는 세션 복원, 원격 기능 플래그, 첫 진입 결정 결과를 조합한다. 계정·PIN·동기화 규칙을 새로 소유하지 않는다.

### 업데이트 안내

`Game`은 업데이트 안내를 생성하되 자동으로 열지 않는 선택지를 제공한다. 시작 조정기가 프로필 선택 완료 후 기존 `maybeShowOnLoad()` 동작을 한 번만 호출한다.

## 8. 오류와 기능 플래그

- `player_profiles_ui`가 꺼져 있으면 필수 선택 화면을 열지 않고 게스트 플레이를 유지한다.
- 원격 설정을 불러오지 못해 프로필 기능을 확인할 수 없으면 게스트 플레이로 풀어주고 게스트 선택을 저장하지 않는다.
- 세션 복원에 실패해도 `게스트로 시작`은 사용할 수 있다.
- 생성·로그인 오류는 기존 일반화된 문구와 시도 제한을 그대로 사용한다.
- 프로필 화면 오류가 업데이트 안내나 게임 초기화를 중단시키지 않는다.
- 두 모달은 동시에 열리지 않는다.

## 9. 디자인·접근성 불변 조건

- 기존 `.player-profile-layer`, 패널 폭, 배경, 모서리, 글자 크기, 버튼 높이, 입력 스타일을 재사용한다.
- 게임 Canvas 크기와 위치, HUD와 무기 바 배치는 바꾸지 않는다.
- 모바일 390x844에서 스크롤 없이 첫 선택 세 버튼이 보이게 한다.
- 모든 행동은 실제 `button`이며 최소 높이 44px을 유지한다.
- 첫 포커스는 제목 다음의 첫 행동 버튼으로 이동한다.
- 모달 안 포커스 순환, 화면 읽기 제목, 상태 알림을 유지한다.
- 색, 소리, 진동만으로 선택 결과를 전달하지 않는다.
- 사용자 문구에 기술 용어와 em dash를 사용하지 않는다.

## 10. 테스트와 승인 기준

### 자동 테스트

- 선택 기록 없음 + 세션 없음 -> `choose`
- 게스트 기록 + 세션 없음 -> `guest`
- 게스트 기록 + 유효한 세션 -> `player`, 게스트 키 삭제
- 기능 플래그 꺼짐 또는 원격 설정 실패 -> 저장하지 않는 게스트 폴백
- 필수 상태의 닫기·배경·Escape·브라우저 뒤로 가기 차단
- 일반 프로필 화면의 기존 닫기 동작 불변
- 게스트 선택 저장 성공·실패
- 생성·로그인 성공 후 필수 상태 종료
- 로그아웃 성공 후 키 삭제와 필수 선택 재표시
- 강제 PIN 변경이 첫 진입 선택보다 우선
- 업데이트 안내가 선택 완료 전에는 열리지 않고 완료 후 한 번만 열림
- 기존 프로필·로그인·동기화 테스트 전체 회귀 통과

### 화면 확인

- 390x844 모바일 프리뷰에서 기존 게임 디자인 변화 없음
- 첫 방문에 시작 방식 선택이 가장 먼저 보임
- 게스트 재방문은 선택 화면 없이 게임 시작
- 기록책에서 여는 프로필 화면은 기존과 동일
- 키보드 포커스, 화면 읽기 이름, 뒤로 가기 동작 확인

### 배포 전 확인

- `npm run lint:copy`
- `npm test`
- `npm run typecheck`
- `npm run build`
- 운영 의존성 보안 감사
- 프리뷰에서 PM이 첫 방문·재방문·로그아웃 흐름과 디자인 불변 확인
- 별도 운영 배포 승인 후 GitHub Pages 배포

## 11. 롤백

DB와 Supabase 변경이 없으므로 데이터 롤백은 필요 없다.

1. 긴급 시 `player_profiles_ui`를 끄면 기존 게스트 플레이로 돌아간다.
2. 프론트 문제가 있으면 GitHub Pages를 직전 성공 배포로 되돌린다.
3. `btw.profileEntry.v1`은 게스트 선택값 하나만 가지므로 남아 있어도 기존 기록이나 로그인 세션에 영향을 주지 않는다.
