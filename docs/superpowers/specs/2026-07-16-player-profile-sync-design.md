# 세상 부수기 플레이어 프로필·기기 동기화 설계

> 작성일: 2026-07-16
>
> 상태: 제품·보안·배포 방향 승인 완료, 문서 검토 대기
>
> 제품 범위: 개인·비상업용 모바일 웹게임

## 1. 결정 요약

게스트의 즉시 플레이를 유지하면서, 원하는 이용자만 `프로필 ID + 숫자 6자리 PIN`으로 프로필을 만든다. 프로필 ID가 로그인 ID이며 전역에서 하나만 존재한다. 새 프로필은 언제나 기록 0부터 시작하고 게스트 기록은 가져오지 않는다.

로그인한 프로필은 게임 기록과 설정 전체를 여러 휴대폰에서 이어 쓴다. 마지막 휴대폰의 전체 저장본으로 덮어쓰지 않고, 의미 있는 변경 내역을 서버가 한 번씩만 반영해 동시 사용·오프라인·재전송에서도 기록이 줄어들지 않게 한다.

사용자 화면에는 이메일이나 Supabase 용어를 노출하지 않는다. Supabase Auth에는 프로필 ID와 무관한 무작위 내부 계정을 만들고, Edge Function이 프로필 ID를 내부 계정으로 안전하게 연결한다. PIN 저장과 세션 발급은 Supabase Auth에 맡긴다.

이번 범위에는 프로필 생성·로그인·로그아웃·동기화 상태·운영자 PIN 재설정·비활성화·삭제를 포함한다. 완성형 마이페이지, 프로필 ID 변경, 게스트 기록 가져오기, 업적 프로필 꾸미기는 다음 단계다.

이 문서는 `2026-07-16-gamification-character-variety-design.md`의 제외 범위 중 `플레이어 회원가입과 기기 간 진행 동기화` 한 항목을 이후 승인으로 대체한다. 그 문서의 조작·캐릭터·진행·운영 원칙은 그대로 유지한다.

## 2. 확정된 제품 원칙

1. 게스트는 가입 안내에 막히지 않고 즉시 플레이한다.
2. 게스트 기록은 현재 기기에만 저장한다.
3. 새 프로필은 기록과 설정을 모두 기본값에서 시작한다.
4. 게스트 기록은 새 프로필에 자동·수동으로 합치지 않는다.
5. 로그아웃하면 그 기기에 남아 있던 게스트 기록으로 돌아간다.
6. 로그인한 프로필의 모든 진행과 설정은 여러 기기에서 동기화한다.
7. 프로필 ID는 로그인 ID이며 중복될 수 없다.
8. PIN은 숫자 정확히 6자리다.
9. 같은 브라우저에서는 로그인 상태를 유지한다.
10. 네트워크·인증·동기화 장애가 게임 플레이를 막지 않는다.

## 3. 목표와 제외 범위

### 목표

- 첫 방문자는 프로필 생성 없이 바로 첫 타겟을 부순다.
- 기록책에서 자연스럽게 프로필 기능을 발견한다.
- 사용자는 프로필 생성 전에 ID 사용 가능 여부를 직접 확인한다.
- 한 기기에서 로그아웃·재로그인해도 게스트와 프로필 기록이 섞이지 않는다.
- 두 기기에서 동시에 플레이해도 카운터·업적·도전 기록이 유실되지 않는다.
- 운영자는 개발자 도움 없이 플레이어 PIN을 재설정하고 프로필을 비활성화·삭제한다.
- 향후 업적·칭호·원형 프로필 꾸미기를 기존 ID 체계 위에 확장한다.

### 이번 범위에서 제외

- 게스트 기록을 프로필로 가져오기
- 프로필 ID 변경
- 이메일·전화번호를 이용한 자동 PIN 찾기
- 사용자의 직접 프로필 삭제 화면
- 공개 프로필 검색, 친구, 순위표, 대전
- 완성형 마이페이지와 업적 프로필 꾸미기 선택
- 소셜 로그인, 이메일 로그인, 전화번호 로그인
- 자체 JWT·세션·비밀번호 해시 시스템

## 4. 페르소나 협의회

### 참여 관점

- 대표·사업·기획·브랜드·마케팅·그로스·CS·운영
- UI/UX 개발·제품 디자인·브랜드 디자인·접근성·모바일 사용성
- 데이터·백엔드·보안·인프라·QA·신뢰성·개인정보

### 1라운드: 독립 검토

모든 관점이 게스트 즉시 플레이, 기록책 상단 프로필 카드, 전용 전체 화면 프로필, 명시적 중복 확인에 합의했다. 데이터·보안 관점은 프로필 이름을 내부 관계키로 쓰지 않고 불변 UUID를 사용해야 한다고 결론 냈다.

주요 반대 의견은 두 가지였다.

1. 이용자가 1~3명이라면 전체 상태를 마지막 저장본으로 덮어써도 충분하다는 의견
2. 6자리 PIN은 가능한 조합이 적어 시도 제한 없이 공개 로그인에 쓰기 위험하다는 의견

첫 번째 의견은 두 휴대폰 동시 사용 요구와 충돌해 기각했다. 두 번째 의견은 PIN을 늘리는 대신 서버 시도 제한, 일반화된 오류, 내부 무작위 계정, 세션 폐기를 필수 조건으로 채택했다.

### 2라운드: 대안 공격·방어

| 안 | 장점 | 결정적 약점 | 판정 |
|---|---|---|---|
| A. 브라우저 직접 로그인 + 전체 상태 마지막 저장 우선 | 가장 빠르고 단순함 | 늦게 도착한 저장이 최신 기록을 덮어쓸 수 있고 로그인 시도 제한을 우회하기 쉬움 | 제외 |
| B-core. Edge 인증 중계 + 의미 단위 변경 동기화 | 내부 인증을 숨기고 중복·역순·오프라인을 안전하게 처리 | 초기 설계·테스트 범위가 큼 | 채택 |
| C. 자체 인증·세션 구축 | 화면을 완전히 맞춤화할 수 있음 | 인증·세션·보안 사고 책임을 제품이 직접 부담 | 제외 |

### 3라운드: 최종 채점

5점 만점이며 정확성과 보안은 기술 채점에서 더 높은 가중치를 적용했다.

| 평가 그룹 | A | B-core | C |
|---|---:|---:|---:|
| 제품·운영 | 2.88 | **4.26** | 3.28 |
| UX·접근성 | 2.30 | **4.60** | 1.80 |
| 데이터·보안·QA | 2.43 | **4.28** | 3.28 |
| 세 그룹 평균 | 2.54 | **4.38** | 2.79 |

최종 결론은 B-core 조건부 Go다. 기능 플래그, 로그인 공격 제한, 동기화 멱등성, 두 기기 실기기 테스트, 관리자 재설정 후 전체 세션 폐기를 운영 배포 조건으로 둔다.

## 5. 화면 정보 구조

### 진입점

- 새 상단 버튼이나 시작 팝업을 추가하지 않는다.
- 기존 `📖 기록책`의 가장 위에 프로필 카드를 둔다.
- 카드 전체를 최소 높이 64px의 실제 버튼으로 만든다.
- 카드를 누르면 기록책 위에 중첩 시트를 열지 않고 전용 전체 화면 `프로필`을 연다.
- 프로필 화면이 열리면 Canvas 입력을 막고 닫을 때 원래 카드로 포커스를 돌린다.

### 게스트 카드

- 제목: `게스트로 즐기는 중`
- 설명: `프로필을 만들면 새 기록을 다른 기기에서도 이어갈 수 있어요.`
- 게스트의 플레이를 중단시키는 자동 가입 팝업은 표시하지 않는다.

### 로그인 카드

- 기본 원형 이미지
- 프로필 ID
- 상태: `저장됨`, `저장하는 중`, `연결되면 저장`, `다시 저장`
- 로그인 성공과 기록 동기화 완료는 서로 다른 상태로 보여준다.

### 기본 원형 이미지

- 프로필 ID의 첫 글자와 내부 UUID에서 결정한 색을 조합한다.
- 같은 프로필은 모든 기기에서 같은 모습이다.
- 사용자가 고르는 UI는 이번에 만들지 않는다.
- 향후 업적 프로필 테두리는 원 바깥에 추가할 수 있게 구성한다.

## 6. 프로필 생성·로그인 흐름

### 프로필 생성

1. 프로필 ID 입력
2. 형식 검사
3. `중복 확인`
4. 사용 가능 상태 확인
5. 숫자 6자리 PIN 입력
6. PIN 재입력
7. 저장 안내와 만 14세 이상 확인
8. 최종 생성
9. 새 프로필 기본 상태를 내려받고 로그인 완료

ID를 수정하면 이전 중복 확인 결과는 즉시 무효화한다. 중복 확인은 이름을 예약하지 않는다. 최종 생성 요청에서 데이터베이스 고유 제약을 다시 검사해 동시에 같은 이름을 생성하는 경쟁도 막는다.

고정 안내 문구:

`새 프로필에서 첫 기록부터 새로 쌓아요. 지금 게스트 기록은 이 기기에 그대로 남아요.`

상태 문구:

- 확인 중: `ID를 확인하는 중이에요`
- 사용 가능: `사용할 수 있는 ID예요`
- 중복: `이미 사용 중인 ID예요. 다른 ID를 골라 주세요.`
- 연결 없음: `인터넷에 연결되면 ID를 확인할 수 있어요.`

### 로그인

1. 프로필 ID와 PIN 입력
2. Edge Function이 시도 제한 검사
3. 무작위 내부 Auth 계정으로 로그인
4. 받은 Supabase 세션을 브라우저에 저장
5. 프로필별 기기 캐시를 먼저 표시
6. 서버 상태와 미전송 변경을 동기화

실패 문구는 존재하지 않는 ID, 잘못된 PIN, 비활성화 상태를 구분하지 않는다.

`ID 또는 PIN을 다시 확인해 주세요.`

### PIN 입력 접근성

- PIN은 6개 칸이 아니라 하나의 실제 `type="password"` 입력으로 만든다.
- `inputmode="numeric"`으로 숫자 키패드를 부른다.
- 붙여넣기를 허용하고 자동 제출하지 않는다.
- `PIN 보기` 버튼을 제공한다.
- 입력 글자 크기는 16px 이상으로 iOS 자동 확대를 피한다.
- 오류는 필드 가까이에 텍스트로 표시하고 실시간 알림 영역으로 읽는다.

## 7. 프로필 ID 계약

- 입력 표기: `display_name`
- 중복 비교키: `name_key`
- 허용 문자: 완성형 한글, 영문 대소문자, 숫자
- 길이: 유니코드 코드포인트 기준 2~12자
- 공백, 기호, 이모지, 자모 단독 입력은 허용하지 않는다.
- 서버에서 NFC 정규화한다.
- ASCII 영문만 소문자로 바꿔 `name_key`를 만든다.
- `Yejin`과 `yejin`은 같은 이름이다.
- 데이터베이스 `UNIQUE(name_key)`가 최종 중복 판정자다.
- 한글 IME 조합 중에는 확정 전 오류를 표시하지 않는다.

논리 검증식은 `^[가-힣A-Za-z0-9]{2,12}$`다. 실제 구현은 브라우저와 Edge Function에서 같은 공유 계약을 사용하고 데이터베이스가 길이·형식·고유성을 다시 확인한다.

## 8. 인증·PIN·세션

### 인증 구조

```text
프로필 ID + PIN
  → 공개 Player Auth Edge Function
  → 형식·시도 제한·기능 플래그 확인
  → 비공개 alias 테이블에서 무작위 Auth 이메일 조회
  → Supabase Auth 비밀번호 로그인
  → access token + refresh token 반환
  → 브라우저 Supabase 세션 저장
```

- 내부 Auth 이메일은 UUID 기반 난수이며 프로필 ID에서 계산하지 않는다.
- 내부 Auth 이메일은 화면·분석·일반 로그에 남기지 않는다.
- PIN은 앞자리 0을 보존하는 문자열로 받고 `^\d{6}$`를 적용한다.
- 별도 PIN 해시 테이블을 만들지 않는다. Supabase Auth가 비밀번호를 저장·검증한다.
- Supabase의 일반 사용자 가입은 계속 닫고, 검증된 Player Auth Edge Function만 서버의 `admin.createUser`로 내부 계정을 만든다.
- ID 확인·생성·로그인은 아직 사용자 JWT가 없는 요청이므로 publishable key를 받은 공개 인증 진입점에서 처리하되, 함수가 요청 형식·기능 플래그·시도 제한을 모두 검사한다.
- service-role 또는 secret key는 Edge Function 환경에만 두고 브라우저 번들에 넣지 않는다.

### 세션

- 브라우저 Supabase 클라이언트의 `persistSession`과 자동 갱신을 사용한다.
- 일반 로그아웃은 `local` 범위로 현재 기기 세션만 종료한다.
- 같은 프로필의 여러 기기 세션을 허용한다.
- 정상 로그아웃 전 미전송 변경을 먼저 보낸다.
- 전송하지 못한 변경이 있으면 `이 기기에 보관하고 로그아웃` 또는 `계속 저장하기`를 선택하게 한다.
- 보관한 프로필 대기열은 다른 프로필에 합치지 않고 같은 프로필 재로그인 때만 재개한다.

### 시도 제한

| 동작 | 제한 |
|---|---|
| 동일 `name_key` + 접속 파생값 로그인 | 15분에 5회 |
| 동일 접속 파생값 전체 로그인 | 1시간에 20회 |
| 프로필 생성 | 동일 접속 파생값 기준 1시간에 5회 |
| ID 중복 확인 | 동일 접속 파생값 기준 1분에 30회 |
| 동기화 | 사용자 기준 1분에 60회, 요청당 최대 100개·256KB |

- 한도를 넘으면 동일한 일반 오류와 재시도 가능 시간을 제공한다.
- 계정을 영구 잠그지 않는다.
- 접속 원문은 분석에 남기지 않는다. 서버 비밀값과 날짜를 이용한 파생값만 짧게 보관한다.
- ID 중복 확인은 제품 요구상 사용 여부를 알려준다. 이 외 로그인·재설정 API는 계정 존재 여부를 드러내지 않는다.

## 9. 데이터 소유권과 서버 스키마

내부 관계는 변경 가능한 프로필 이름이 아니라 `auth.users.id` UUID를 사용한다.

### 주요 테이블

| 테이블 | 역할 |
|---|---|
| `player_profiles` | UUID, 표시 ID, 중복키, 상태, 자격 버전, PIN 변경 필요 상태 |
| `player_auth_aliases` | UUID와 무작위 Auth 이메일 연결, 브라우저 접근 금지 |
| `player_progress` | 계정의 현재 투영 상태, 계정 시드, 스키마·카탈로그 버전, 서버 revision |
| `player_devices` | 기기별 마지막 연속 처리 번호와 마지막 접속 시각 |
| `player_sync_operations` | 중복 방지용 의미 단위 변경 기록 |
| `player_auth_rate_limits` | 로그인·생성·중복 확인의 짧은 제한 창 |
| `admin_audit_logs` | PIN 재설정·비활성화·삭제 작업의 관리자·시각·결과 |

### 권한

- 플레이어 테이블의 브라우저 직접 쓰기를 허용하지 않는다.
- 인증·동기화는 Edge Function과 검증된 데이터베이스 함수만 통과한다.
- 함수는 JWT의 `auth.uid()`를 사용하며 요청 본문의 사용자 UUID를 신뢰하지 않는다.
- custom access-token hook이 현재 `credential_version`을 JWT claim에 넣고, 모든 플레이어 Edge Function이 데이터베이스 현재값과 비교한다.
- 운영자 권한은 기존 `admin_users` 중앙 검사만 사용한다.
- 분석에는 이벤트 enum, 개수, 내부 ID만 남기며 프로필 ID·PIN·포인터 위치를 남기지 않는다.

## 10. 게스트·프로필 저장 분리

### 게스트

- 기존 `btw.progress.v1`을 그대로 사용한다.
- 프로필 생성·로그인 때문에 이동하거나 삭제하지 않는다.
- 서버에 업로드하지 않는다.

### 로그인 프로필

- 내부 UUID별 캐시와 IndexedDB 대기열을 사용한다.
- 계정 시드는 서버가 새 프로필 생성 시 만든다.
- 로그인 직후 서버 상태가 늦어도 캐시된 프로필 화면으로 게임을 시작할 수 있다.
- 로그아웃·다른 프로필 로그인 중 늦게 도착한 응답은 사용자 UUID와 세션 세대가 일치할 때만 반영한다.
- 저장소가 실패하면 메모리로 계속 플레이하고 쉬운 문구로 상태를 알린다.

## 11. 동기화 계약

### 원칙

- 전체 상태의 마지막 저장 우선 덮어쓰기를 금지한다.
- 게임 프레임이나 포인터 이동을 전송하지 않는다.
- 기존 진행 저장 체크포인트에서 의미 단위 변경만 대기열에 추가한다.
- 각 변경은 `user_id + device_id + client_seq + operation_id + operation_version`을 가진다.
- 데이터베이스 고유 제약으로 같은 변경을 정확히 한 번만 반영한다.
- 서버는 사용자별 투영 행을 잠근 하나의 트랜잭션에서 배치와 revision을 갱신한다.
- 중간 번호가 비면 다음 번호부터 적용하지 않고 누락된 번호 재전송을 요청한다.

### 병합 규칙

| 데이터 | 병합 |
|---|---|
| 유효 공격·충전 마무리·누적 파괴·무기 사용·마무리·타겟 파괴 | 기기별 증가분 합산 |
| 최고 연속 | 가장 큰 값 |
| 사용한 무기·본 기술·해금 업적 | 집합 합치기 |
| 업적 해금 시각 | 가장 이른 유효 시각 |
| 업적 확인 상태 | 한 번 확인하면 유지 |
| 도장 수 | 완료한 고유 도전의 합, 중복 지급 없음 |
| 오늘의 도전 진행 | 서버가 의미 이벤트를 현재 계정 도전 규칙에 적용 |
| 오늘의 도전 완료 | 계정·날짜·도전 ID별 한 번만 완료·지급 |
| 칭호·모습·강한 공격 방식·모션·진동 | 항목별 서버 수신 순서의 마지막 변경 |

- 카운터와 업적은 단조 증가하며 동기화 때문에 감소하지 않는다.
- 설정만 사용자의 명시적 최신 선택으로 바뀔 수 있다.
- 원본 동기화 연산은 90일 후 축약할 수 있지만 기기별 마지막 처리 번호와 최종 투영은 유지한다.
- 클라이언트가 보낸 enum·ID·증가량은 현재 카탈로그와 상한으로 검증한다.
- 경쟁 보상이나 순위표가 없으므로 조작된 개인 기록이 다른 사용자에게 영향을 주지 않는다.

### 오프라인과 재시도

- 오프라인에서 게임과 로컬 대기열 기록을 계속한다.
- 연결이 돌아오면 1초, 2초, 4초로 최대 3회 즉시 재시도한다.
- 이후에는 다음 저장 체크포인트와 앱 복귀 때 다시 시도한다.
- 400·401·403은 자동 반복하지 않는다.
- 세션 만료는 한 번 갱신하고 실패하면 로그인 안내로 전환한다.
- 대기열이 커져도 한 번에 100개·256KB 이하로 나눈다.

## 12. 운영자 플레이어 관리

기존 운영자 화면에 `플레이어` 영역을 추가한다.

### 목록

- 프로필 ID
- 상태: 사용 중·비활성
- 생성일
- 마지막 동기화 시각
- PIN 변경 필요 여부
- 원시 진행 데이터와 내부 Auth 이메일은 기본 목록에 표시하지 않는다.

### PIN 재설정

1. 운영자가 기존 개인 연락 방식으로 요청자를 확인한다.
2. 프로필 ID를 검색한다.
3. 새 임시 숫자 6자리 PIN을 두 번 입력한다.
4. 재설정 요청 ID를 만들고 `credential_version`을 올려 기존 access token을 즉시 거부한다.
5. Supabase Auth 비밀번호를 임시 PIN으로 변경한다.
6. 서버가 임시 PIN으로 한 번 로그인해 사용자 JWT를 받은 뒤 `global` 로그아웃으로 기존·방금 생성된 refresh session을 모두 폐기한다.
7. `force_pin_change`를 켜고 재설정을 완료 처리한다.
8. 다음 로그인에서 새 숫자 6자리 PIN 변경을 요구한다.

Auth와 데이터베이스 변경은 하나의 트랜잭션이 아니므로 재설정은 단계별 상태를 가진 멱등 작업으로 만든다. 중간 실패를 성공으로 표시하지 않고 같은 요청 ID로 안전하게 재개한다. 로컬 Supabase 통합 테스트에서 실제 refresh token 폐기와 이전 JWT 거부를 증명하지 못하면 운영 배포하지 않는다.

운영 기록에는 관리자 UUID, 대상 사용자 UUID, 작업 종류, 시각, 성공 여부만 남긴다. PIN은 응답·로그·분석·감사 기록에 남기지 않는다.

### 비활성화·삭제

- 비활성화는 새 로그인과 동기화를 닫되 서버 데이터를 유지한다.
- 삭제는 정확한 프로필 ID 재입력과 두 번째 확인이 필요하다.
- 삭제 시 Auth 사용자 삭제를 기준으로 프로필·진행·기기·동기화 데이터가 cascade 삭제된다.
- 삭제 후에는 복구를 약속하지 않는다.
- 사용자의 직접 삭제 UI와 이름 재사용 유예 정책은 다음 단계에서 별도 설계한다.

## 13. 개인정보와 연령

### 수집 최소화

- 수집: 프로필 ID, 인증 정보, 게임 진행, 설정, 기능 운영에 필요한 짧은 보안 파생값
- 수집하지 않음: 이메일, 전화번호, 실명, 생년월일, 주소, 원시 포인터 좌표
- 이용 목적: 로그인, 기기 간 기록 저장, 운영자 PIN 재설정, 공격 방지
- 보유: 프로필 삭제 전까지, 짧은 보안 제한 데이터는 제한 창 종료 후 정리

프로필 생성 전에 쉬운 개인정보 처리 안내 링크와 다음 확인을 둔다.

`만 14세 이상이며, 프로필과 기록 저장 안내를 확인했어요.`

만 14세 미만은 개인정보를 저장하지 않는 게스트 플레이를 계속 이용할 수 있다. 법정대리인 확인 시스템은 이번 범위에 넣지 않는다. 운영 배포 전 처리 목적·항목·보유 기간·삭제 요청 방법·처리 위탁 정보를 담은 한국어 개인정보 처리 안내를 확정한다.

연령 확인 문구는 법률 자문이나 법정대리인 확인을 대신하지 않는다. 공개 대상이 기존 개인 사용자 범위를 넘어갈 때에는 프로필 가입 연령·동의 방식을 다시 검토한다.

Supabase 프로젝트 리전이 확정되면 처리 위탁·국외 처리에 해당하는 안내 항목을 확인한다. 운영자 삭제 요청을 받을 실제 연락 방법이 개인정보 안내에 들어가기 전에는 프로필 생성 플래그를 열지 않는다.

개인정보 안내 문구는 한 모듈과 버전 상수로 관리한다. 실질 내용이 바뀌면 버전을 올려 다음 프로필 진입 때 다시 확인한다.

## 14. 접근성·모바일 품질

- 모든 조작은 최소 44px, 주요 제출 버튼은 48px 이상이다.
- 색만으로 사용 가능·오류·동기화 상태를 전달하지 않는다.
- 오류는 필드 가까이에 표시하고 요약 포커스를 제공한다.
- 로딩 버튼은 중복 제출을 막고 진행 문구를 가진다.
- 브라우저 뒤로가기와 닫기 버튼이 같은 정리 동작을 수행한다.
- 320px 폭, 390×844, 200% 글자 확대에서 가로 잘림이 없어야 한다.
- `prefers-reduced-motion`에서는 프로필 전환 애니메이션을 줄인다.
- iOS Safari·Android Chrome에서 새로고침, 앱 복귀, 네트워크 전환, 숫자 키패드를 실기기로 확인한다.
- 사용자 확대를 막는 viewport 설정을 제거한다.

## 15. 기능 플래그와 장애 처리

모든 플래그는 운영 배포 전 기본 닫힘이다.

| 플래그 | 닫혔을 때 |
|---|---|
| `player_profiles_ui` | 게스트에게 프로필 카드 미노출, 이미 로그인한 사용자의 안전한 로컬 상태는 유지 |
| `player_signup` | 신규 생성만 닫고 기존 로그인·동기화 유지 |
| `player_sync_writes` | 새 변경을 기기에 대기시키고 게임·기존 상태 읽기는 유지 |

- 장애 시 신규 가입부터 닫는다.
- 동기화 장애는 로컬 대기열로 흡수한다.
- 데이터베이스 마이그레이션을 되돌려 사용자 데이터를 파괴하지 않는다.
- 서버는 최소 이전 연산 버전 N-1을 지원해 오래 열린 탭의 요청을 흡수한다.

## 17. 단계별 배포와 롤백

1. 데이터베이스·Edge Function을 먼저 배포하고 세 플래그를 닫아둔다.
2. 운영자 전용 테스트 프로필로 생성·로그인·두 기기 동기화·재설정·삭제를 검증한다.
3. 게임·운영자 미리보기 URL에서 모바일 화면과 기존 21개 무기 회귀를 검증한다.
4. PM이 새 미리보기를 확인하고 운영 배포를 승인한다.
5. 정적 번들을 운영 배포한다. 기존 사용자는 아직 게스트 경험을 유지한다.
6. `player_profiles_ui`, `player_signup`, `player_sync_writes` 순서로 연다.
7. 실제 URL에서 가입·로그인·동기화·운영자 화면을 다시 확인한다.
8. 오류율, 로그인 실패, 대기열 지연, 예상 동기화 이벤트 부재를 최소 5분 관찰한다.

롤백은 플래그를 닫아 신규 작업을 멈추고 로그인 사용자의 변경을 로컬에 보관하는 방식이다. 이미 생성된 프로필 데이터베이스를 파괴하거나 이전 마이그레이션으로 내리지 않는다.

## 18. 향후 마이페이지 확장

현재 프로필 화면이 향후 마이페이지의 최상위 경로가 된다. 이번에 불변 사용자 UUID, 업적 ID, 선택 칭호, 캐릭터 모습 데이터를 유지하므로 다음 단계에서 아래를 추가할 수 있다.

- 업적별 상세 기록과 획득일
- 대표 칭호 선택·전시
- 업적으로 받은 원형 프로필 이미지
- 업적 프로필 테두리·꾸미기
- 선택한 꾸미기의 기기 간 동기화

이번에는 미래 필드나 빈 메뉴를 미리 노출하지 않는다. 기본 원형 이미지는 계산값으로 만들고, 실제 업적 프로필 선택이 생길 때 nullable 장착 ID와 카탈로그를 별도 migration으로 추가한다.

## 19. 구현 계획 분할

문서 승인 뒤 다음 세 계획을 순서대로 작성하고 각각 검증한다.

1. **Auth·데이터 기반**: migration, Edge 인증 중계, ID 계약, 시도 제한, 세션, 관리자 재설정·삭제
2. **프로필 UX·저장 분리**: 기록책 카드, 전체 화면 프로필, 생성·로그인·로그아웃, 게스트·계정 캐시 분리, 접근성
3. **동기화·운영·배포**: 멱등 대기열, 서버 투영, 운영자 플레이어 화면, 다중 기기 QA, 기능 플래그, 미리보기·운영 배포

각 계획은 TDD로 실패 테스트를 먼저 만들고, 구현·로컬 검증·적대적 검토를 통과한 다음 다음 계획으로 넘어간다.

## 20. 공식 근거

- Supabase Auth 세션은 access token과 refresh token으로 구성되며 여러 기기 세션과 세션 제한을 지원한다: <https://supabase.com/docs/guides/auth/sessions>
- 브라우저 Supabase 클라이언트는 기본적으로 세션 지속 저장을 지원한다: <https://supabase.com/docs/reference/javascript/auth>
- Supabase 로그인 오류는 계정 없음과 잘못된 비밀번호를 구분하지 않는 응답을 지원한다: <https://supabase.com/docs/reference/javascript/auth-signinwithpassword>
- Supabase 로그아웃은 `local`, `global`, `others` 범위를 지원한다: <https://supabase.com/docs/guides/auth/signout>
- 서버의 Auth 사용자 생성·비밀번호 변경은 브라우저에 service-role을 노출하지 않고 수행해야 한다: <https://supabase.com/docs/reference/javascript/auth-admin-createuser>, <https://supabase.com/docs/reference/javascript/auth-admin-updateuserbyid>
- 인증된 Edge Function은 사용자 JWT와 RLS 문맥을 사용할 수 있다: <https://supabase.com/docs/guides/functions/auth>
- 만 14세 미만 아동의 개인정보 처리 동의에는 법정대리인 동의 확인이 필요하다: <https://www.law.go.kr/LSW/lsSideInfoP.do?docCls=jo&joBrNo=02&joNo=0022&lsiSeq=270351&urlMode=lsScJoRltInfoR>
